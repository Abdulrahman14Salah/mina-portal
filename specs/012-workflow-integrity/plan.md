# Implementation Plan: Workflow Integrity Fixes

**Branch**: `012-workflow-integrity` | **Date**: 2026-03-22 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/012-workflow-integrity/spec.md`

## Summary

Fix two logic bugs in `WorkflowService`: (1) the `workflow_started` audit log fires even when no tasks are seeded due to a `return` inside a closure not exiting the outer method; (2) the next-task lookup uses `position = current + 1` which silently fails when task positions have gaps. Both fixes are confined to `app/Services/Tasks/WorkflowService.php` with a new test file. No schema changes, no new routes, no Blade changes.

## Phase 1: Design

### Data Model
See [data-model.md](data-model.md).

- **No database migrations**
- `application_tasks.position` is an existing integer column; positions are not guaranteed contiguous
- Audit log entry for `workflow_started` must only exist when `$seeded === true`

### Interface Contracts
No external interface contracts needed — this is a purely internal service fix. No HTTP endpoints, routes, or API signatures change.

### Implementation Strategy

**Two independent fixes, both in the same file:**

#### Fix 1 — Audit Log Guard (`seedTasksForApplication`)

Current code (broken):
```php
DB::transaction(function () use ($application, $hasSections): void {
    // ...
    if ($templates->isEmpty()) {
        return;  // Only exits closure — auditLog still fires
    }
    // ...
});
$this->auditLog->log('workflow_started', ...);  // Always runs — BUG
```

Correct pattern:
```php
$seeded = false;

DB::transaction(function () use ($application, $hasSections, &$seeded): void {
    // ...
    if ($templates->isEmpty()) {
        return;  // Exits closure; $seeded remains false
    }
    // ...
    if (! empty($tasks)) {
        $tasks[0]->update(['status' => 'in_progress']);
        $application->update(['status' => 'in_progress']);
        $seeded = true;  // ← Mark as seeded
    }
});

if ($seeded) {
    $this->auditLog->log('workflow_started', ...);  // Only fires when tasks exist
}
```

#### Fix 2 — Next-Task Query (`advanceTask` and `approveTask`)

Current code (broken for gaps):
```php
$nextTask = ApplicationTask::where('application_id', $task->application_id)
    ->where('position', $task->position + 1)  // BUG: assumes contiguous positions
    ->first();
```

Correct pattern (same replacement in both methods):
```php
$nextTask = ApplicationTask::where('application_id', $task->application_id)
    ->where('position', '>', $task->position)
    ->orderBy('position')
    ->first();
```
