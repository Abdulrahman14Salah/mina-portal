# Implementation Plan: Menu Unification — Foundation and Architecture

**Branch**: `011-menu-unification-foundation` | **Date**: 2026-03-22 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/011-menu-unification-foundation/spec.md`

## Summary

Replace the three separate, inconsistent navigation implementations (config-driven admin nav, inline-array client tabs, absent reviewer nav) with a single `config/navigation.php` catalogue filtered at runtime by `MenuService`. A `MenuComposer` injects `$menuItems` into all nav Blade components. Role-specific layout templates remain separate; each includes the shared `<x-nav.main />` component. All labels are translation-key driven (EN + AR). No database changes.

## Phase 0: Research Findings

All decisions resolved. See [research.md](research.md) for full rationale.

| Decision | Choice |
|----------|--------|
| Item storage | PHP config (`config/navigation.php`) |
| Role filtering | `MenuService::getForUser(User $user): array` |
| View injection | `MenuComposer` (View Composer pattern, matches `AdminBreadcrumbComposer`) |
| Active state | `request()->routeIs($item['active_pattern'] ?? $item['route'])` |
| Translation keys | Dedicated `resources/lang/{locale}/navigation.php` |
| Layout strategy | Keep 3 layouts; each includes shared `<x-nav.main />` |
| Client tabs | Included in unified config with `route_params` |

## Phase 1: Design

### Data Model
See [data-model.md](data-model.md).

- **No database migrations**
- `config/navigation.php`: 17 items across 3 roles; each item has `label_key`, `route`, `route_params`, `roles`, `active_pattern`, `position`
- `resources/lang/en/navigation.php` + `resources/lang/ar/navigation.php`: 18 translation keys

### Interface Contracts
See [contracts/menu-service.md](contracts/menu-service.md).

- `MenuService::getForUser(User $user): array` — filtered, position-sorted items
- `MenuComposer::compose(View $view)` — injects `$menuItems` before render
- `<x-nav.main />` — renders `$menuItems` as `<a>` links with active state

### Implementation Strategy

**Phase order (independent increments)**:

1. **Foundation** (no UI change yet): Create `config/navigation.php`, `MenuService`, `MenuComposer`, lang files. Register composer. Existing pages unaffected.

2. **Admin nav** (lowest risk, already partially done): Update `components/admin/nav.blade.php` to use `$menuItems`. Verify admin nav still renders identically.

3. **Reviewer nav** (new capability): Create `components/reviewer/nav.blade.php`. Update `reviewer-layout.blade.php` to include it. Reviewer now has a proper nav.

4. **Client nav** (refactor existing): Update `components/client/nav.blade.php` to use `$menuItems` from composer. Remove inline array.

5. **Cleanup**: Remove `config/admin-navigation.php`. Run full test suite.

6. **Tests**: `MenuServiceTest` (filtering logic, edge cases) + `MenuVisibilityTest` (role-based HTTP tests).

Each step is independently deployable and testable. Steps 2–4 can be executed in any order after step 1 completes.
